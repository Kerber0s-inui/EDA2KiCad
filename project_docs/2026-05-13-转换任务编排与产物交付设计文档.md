# 转换任务编排与产物交付设计文档
## 1. 背景

当前项目已经具备以下能力：

- 多种单文件转换策略：`custom`、`kicad-official`、`kicad-gui-official`、`third-party`
- 统一调用入口：`ConversionService`、CLI、Web
- GUI/CLI/第三方策略可在内部创建中间目录并产出最终 KiCad 文件

但当前交付形态仍有几个问题：

- 用户未指定输出目录时，系统仍可能使用内部默认目录
- 最终输出目录直接混入中间文件、项目文件和调试文件，不利于一次任务的独立读取
- Web 只支持单文件上传，不支持上传包含 `PrjPcb`、`PcbDoc`、`SchDoc` 的 ZIP
- 不同输入组合下是否新建空工程、是否复用 `PrjPcb`，当前没有统一编排规则

本设计文档定义一个统一的“转换任务编排层”，用于规范输入解析、输出目录组织、中间产物清理和 ZIP 联合转换。

## 2. 目标

### 2.1 功能目标

- 强制要求用户显式提供 `output_dir`
- 每次转换都在用户指定目录下创建一个独立任务子目录
- 单文件和 ZIP 均通过统一编排层执行
- 支持以下输入组合：
  - `PrjPcb + PcbDoc + SchDoc`
  - `PrjPcb + PcbDoc`
  - `PrjPcb + SchDoc`
  - `PcbDoc + SchDoc`
  - 仅 `PrjPcb`
  - 仅 `PcbDoc`
  - 仅 `SchDoc`
- 最终任务目录只保留本次交付所需产物：
  - `.kicad_pcb`
  - `.kicad_sch`
  - `.kicad_pro`
  - `report.json`

### 2.2 非功能目标

- 中间目录必须在任务完成后清理
- 保持现有策略层尽量不感知 ZIP 和复杂输入编排
- Web、CLI、后续云端 API 共用同一套任务编排逻辑
- 最终任务目录应便于 AI 智能体直接读取

### 2.3 非目标

- 本轮不实现 ZIP 内多工程、多板、多原理图的复杂自动配对
- 本轮不做 GUI 自动化流程重写，只在现有能力上增加任务编排
- 本轮不实现新的转换策略，只编排已有策略

## 3. 核心规则

### 3.1 输出目录规则

- `output_dir` 为必填项
- 若未提供，CLI/Web/API 直接报错
- 每次任务在 `output_dir` 下创建一个唯一子目录
- 子目录命名规则：
  - `<YYYYMMDD_HHMMSS>_<input_label>`
  - 例如：`20260513_231045_demo`

### 3.2 最终保留产物

任务目录中只保留以下文件：

- `*.kicad_pcb`
- `*.kicad_sch`
- `*.kicad_pro`
- `report.json`

必须清理的内容包括但不限于：

- `.eda2kicad/`
- GUI job 工作目录
- 临时解压目录
- 临时导入工程目录
- `-cache.lib`
- 截图、窗口 dump、自动化日志
- 未最终交付的中间 `.kicad_pro`

说明：

- 若本次只转换 PCB，则最终目录中不要求存在 `.kicad_sch`
- 若本次只转换原理图，则最终目录中不要求存在 `.kicad_pcb`
- `.kicad_pro` 保留一份最终工程文件

### 3.3 ZIP 输入规则

- 支持上传单个 ZIP 文件
- ZIP 中可包含：
  - `.PrjPcb`
  - `.PcbDoc`
  - `.SchDoc`
- ZIP 中若只包含其中一种或两种文件，也应支持对应转换
- ZIP 解压后只允许形成唯一的一组候选输入；如果同类文件存在多个且无法唯一判定，直接报错

唯一判定规则：

- 每类文件只有一个时，直接使用
- 如果 `PrjPcb`、`PcbDoc`、`SchDoc` 主文件名完全一致，则视为一组
- 超出上述范围的多文件情况，本轮不猜测，直接失败

## 4. 输入组合与工程复用规则

### 4.1 `PrjPcb + PcbDoc + SchDoc`

- 复用 `PrjPcb` 对应工程上下文
- 不新建空工程
- 输出：
  - `.kicad_pcb`
  - `.kicad_sch`
  - `.kicad_pro`
  - `report.json`

### 4.2 `PrjPcb + PcbDoc`

- 复用 `PrjPcb`
- 不新建空工程
- 输出：
  - `.kicad_pcb`
  - `.kicad_pro`
  - `report.json`

### 4.3 `PrjPcb + SchDoc`

- 复用 `PrjPcb`
- 不新建空工程
- 输出：
  - `.kicad_sch`
  - `.kicad_pro`
  - `report.json`

### 4.4 `PcbDoc + SchDoc`

- 在同一个空工程下完成两者转换
- 只创建一份空 `.kicad_pro`
- PCB 与原理图都归入这个工程
- 输出：
  - `.kicad_pcb`
  - `.kicad_sch`
  - `.kicad_pro`
  - `report.json`

### 4.5 仅 `PcbDoc`

- 新建空工程
- 只转换 PCB

### 4.6 仅 `SchDoc`

- 新建空工程
- 只转换原理图

### 4.7 仅 `PrjPcb`

- 尝试复用该工程进行导入
- 如果工程内部可唯一定位到可导入对象，则执行转换
- 若无法唯一定位，则报错，不猜测

## 5. 编排层设计

### 5.1 新增职责

新增一个任务编排层，负责：

- 校验输入和 `output_dir`
- 解析单文件或 ZIP
- 生成任务目录
- 推导输入组合
- 决定“复用 `PrjPcb`”还是“新建空工程”
- 调用现有 `ConversionService` 或更底层策略
- 汇总最终产物
- 清理所有中间目录
- 生成统一 `report.json`

### 5.2 推荐模块划分

- `src/eda2kicad/jobs/models.py`
  - 定义任务输入、任务目录、候选文件集合、任务结果
- `src/eda2kicad/jobs/planner.py`
  - 解析输入、识别文件组合、确定任务模式
- `src/eda2kicad/jobs/workspace.py`
  - 创建任务子目录和临时目录，负责清理
- `src/eda2kicad/jobs/runner.py`
  - 协调原理图/PCB 子任务执行
- `src/eda2kicad/jobs/reporting.py`
  - 生成统一 `report.json`

### 5.3 与现有层的边界

- Strategy 继续负责“给定一个输入，产出一类 KiCad 文件”
- 任务编排层负责“一个请求里有哪些输入、需要输出哪些最终产物”
- Web/CLI 不再自己处理 ZIP、任务目录或中间文件清理

## 6. 统一 report 结构

保留一个统一的 `report.json`，建议结构如下：

```json
{
  "job": {
    "job_id": "20260513_231045_demo",
    "input_mode": "zip",
    "strategy": "kicad-gui-official",
    "created_at": "2026-05-13T23:10:45+08:00"
  },
  "inputs": {
    "prjpcb": "demo.PrjPcb",
    "pcbdoc": "demo.PcbDoc",
    "schdoc": "demo.SchDoc"
  },
  "outputs": {
    "project": "demo.kicad_pro",
    "board": "demo.kicad_pcb",
    "schematic": "demo.kicad_sch"
  },
  "runs": {
    "board": {},
    "schematic": {}
  },
  "cleanup": {
    "intermediate_deleted": true
  }
}
```

说明：

- `runs.board` 和 `runs.schematic` 中分别写入对应策略的原始 `report`
- 编排层只增加外层任务信息和清理结果

## 7. 异常处理规则

- 未提供 `output_dir`：直接报错
- ZIP 中没有受支持文件：直接报错
- ZIP 中存在多个同类文件且无法唯一判定：直接报错
- 某个子任务失败：
  - 默认整次任务失败
  - 仍保留 `report.json`
  - 不保留半成品交付目录中的无效最终文件
- 中间目录清理失败：
  - 任务转换可视为成功
  - 在 `report.cleanup` 中记录清理告警

## 8. 测试策略

需要覆盖以下场景：

- Web 未提供 `output_dir` 报错
- 单文件 `PcbDoc` 仅输出 `kicad_pcb + kicad_pro + report`
- 单文件 `SchDoc` 仅输出 `kicad_sch + kicad_pro + report`
- `PcbDoc + SchDoc` ZIP 在同一空工程下输出三类文件
- `PrjPcb + PcbDoc` ZIP 复用工程，不新建空工程
- `PrjPcb + SchDoc` ZIP 复用工程，不新建空工程
- `PrjPcb + PcbDoc + SchDoc` ZIP 同时输出四类文件
- ZIP 中同类文件多个且无法唯一判定时报错
- 成功任务结束后中间目录被清理

## 9. 实施建议

建议按以下顺序实施：

1. 先引入任务编排层与任务目录规则
2. 再接通 Web 对 ZIP 的支持
3. 再让 CLI 也复用相同任务层
4. 最后扩展 `kicad-gui-official` 以支持同一空工程下的 `PcbDoc + SchDoc` 联合转换

这样可以先稳定输入和输出边界，再处理最复杂的联合 GUI 导入逻辑。
