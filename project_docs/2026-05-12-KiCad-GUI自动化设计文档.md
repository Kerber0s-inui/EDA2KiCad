# KiCad GUI 自动化设计文档

## 1. 背景与目标

本项目当前已经具备以下几类 Altium 到 KiCad 的转换能力：

- `custom`：自研文本原理图转换与原生 `.SchDoc` 原理图解析
- `kicad-official`：KiCad 官方 CLI 导入能力
- `third-party`：基于 `altium2kicad` 的第三方转换能力

现阶段的核心问题是：

- KiCad 官方 CLI 对部分输入存在崩溃或不可用情况
- 第三方方案对部分文件支持不稳定，尤其是不同文件格式变体
- 某些通过 KiCad GUI 可以成功导入的场景，CLI 无法稳定覆盖

因此，需要新增一条 **KiCad GUI 自动化导入链路**，用于：

- 优先解决 `PCB` 原生导入的自动化问题
- 后续扩展到原理图 GUI 导入
- 从一开始就考虑 **云端 Windows 桌面代理** 与 **API 调用面**
- 保持与现有 `ConversionService / CLI / Web` 的统一调用方式

本设计文档先定义 GUI 自动化的第一阶段实现方向：

- 先实现 `PCB GUI 自动化导入`
- 再扩展 `原理图 GUI 自动化导入`
- 运行环境为 `Windows + 已安装 KiCad + 持续在线桌面会话`
- 后续可封装成 API，供其他 AI 智能体调用

## 2. 设计目标

### 2.1 功能目标

- 支持通过 KiCad GUI 自动导入 `Altium .PcbDoc`
- 自动保存导入结果为 `.kicad_pcb`
- 尽量补齐 `.kicad_pro`
- 保留自动化日志、截图和失败现场
- 通过现有 `ConversionService` 统一暴露
- 可被 CLI、Web、未来 API 共用

### 2.2 非功能目标

- 支持无人值守运行，但必须建立在“可交互桌面会话”基础上
- 导入流程具备阶段感知、失败恢复、超时控制和证据留存
- 单机单桌面会话第一阶段按串行执行，不追求 GUI 并发
- 自动化代码应与业务转换策略解耦，便于后续扩展原理图导入

### 2.3 非目标

本阶段不包含以下内容：

- 纯 headless 无桌面导入
- 多桌面多任务并发调度
- 直接依赖图像识别作为主驱动
- 在 GUI 自动化流程中直接执行 DRC、规则优化和版图质量判定
- 一次性完成 PCB 和原理图两条 GUI 自动化链路

## 3. 总体方案

采用如下主方案：

- **Windows 桌面代理 + Python + pywinauto/UI Automation + 少量图像识别兜底**

同时在接口层预留后续混合编排能力：

- CLI 可用时优先 CLI
- CLI 失败时可切 GUI
- GUI 失败后可选择回退第三方方案

### 3.1 方案选择理由

该方案最符合以下约束：

- 目标运行环境允许 Windows 桌面会话
- 后续要封成 API 给其他 AI 调用
- 需要比坐标点击更高的稳定性和可维护性
- 又要允许在个别控件无法被 UIA 可靠识别时使用图像兜底

### 3.2 统一接入位置

GUI 自动化必须接在现有统一调用面下，而不是单独做脚本入口：

- `ConversionService`
- `CLI`
- `Web`
- 后续 `API`

策略层建议新增：

- `kicad-gui-official`

该策略与现有：

- `custom`
- `kicad-official`
- `third-party`

并列存在，由调用方显式选择，或由后续编排器自动选择。

## 4. 架构设计

### 4.1 分层结构

建议新增以下逻辑分层：

1. `Conversion Interface`
   - 现有 `ConversionService` 保持不变
   - 新增 GUI 自动化策略入口

2. `Strategy Layer`
   - `kicad-gui-official`
   - 负责本次导入任务的策略分发与产物回收

3. `GUI Automation Driver`
   - 负责 KiCad GUI 启动、窗口识别、菜单操作、对话框处理、保存文件
   - 基于 `pywinauto/UIA`
   - 必要时少量图像识别兜底

4. `Session / Runtime Layer`
   - 检查和管理桌面会话
   - 任务互斥
   - 超时管理
   - 截图、日志、错误现场管理

5. `Artifact Collector`
   - 收集 `.kicad_pcb`、`.kicad_sch`、`.kicad_pro`
   - 收集 `report.json`、截图、日志
   - 回填到统一返回结构

### 4.2 推荐模块划分

建议后续新增模块如下：

- `src/eda2kicad/gui/session.py`
- `src/eda2kicad/gui/runtime.py`
- `src/eda2kicad/gui/windows.py`
- `src/eda2kicad/gui/driver.py`
- `src/eda2kicad/gui/pcb_import.py`
- `src/eda2kicad/gui/schematic_import.py`
- `src/eda2kicad/strategies/kicad_gui_official.py`

其中职责建议如下：

- `session.py`
  - 桌面会话检测
  - 任务锁
  - 工作目录初始化

- `runtime.py`
  - 自动化运行时上下文
  - 日志记录
  - 截图记录
  - 阶段状态推进

- `windows.py`
  - 顶层窗口枚举
  - 窗口等待
  - 弹窗识别

- `driver.py`
  - KiCad GUI 通用操作
  - 启动程序
  - 菜单点击
  - 对话框填值
  - 文件保存

- `pcb_import.py`
  - PCB GUI 导入阶段机

- `schematic_import.py`
  - 原理图 GUI 导入阶段机

- `kicad_gui_official.py`
  - GUI 官方策略封装
  - 与 `ConversionPayload` 对齐

## 5. PCB GUI 自动化流程

### 5.1 主流程

第一阶段只实现 PCB GUI 自动化，主流程固定为：

1. 预检查
2. 启动 KiCad
3. 等待主窗口可用
4. 打开 PCB 导入入口
5. 选择 `.PcbDoc`
6. 确认导入
7. 等待导入完成
8. 保存 `.kicad_pcb`
9. 生成或补齐 `.kicad_pro`
10. 收集产物与日志

### 5.2 阶段机设计

PCB 导入建议拆成明确阶段：

- `launch_kicad`
- `wait_main_window`
- `open_import_menu`
- `select_input_file`
- `confirm_import`
- `wait_import_complete`
- `save_output`
- `collect_artifacts`

每个阶段都应具备：

- 进入条件
- 完成条件
- 超时时间
- 重试上限
- 失败现场记录

### 5.3 等待机制

禁止用纯固定睡眠作为主等待逻辑，应该优先使用条件等待：

- 目标窗口出现
- 菜单项可点击
- 文件对话框出现
- 导入完成后编辑窗口可交互
- 保存成功后目标文件存在

固定短睡眠只允许作为状态切换后的最小缓冲。

### 5.4 产物输出

导入成功后统一写入：

- `<project>.kicad_pcb`
- `<project>.kicad_pro`（若未生成则补齐最小工程文件）
- `report.json`

附加产物：

- `automation.log`
- `screenshots/*.png`
- `window-dump.json`

### 5.5 与现有后处理衔接

GUI 自动化只负责“导入并保存”。

其后可继续沿用现有服务中的后处理能力，例如：

- 工程文件补齐
- 规则写回
- 后续质量检查

这样 GUI 自动化与“设计规则覆写”保持解耦。

## 6. 原理图 GUI 自动化扩展

第二阶段复用同一套 GUI 自动化底座，增加原理图导入。

### 6.1 复用部分

- KiCad 启动
- 桌面会话检测
- 窗口等待
- 菜单与对话框操作
- 截图与日志
- 产物收集

### 6.2 新增部分

- 进入原理图编辑器或原理图导入入口
- 选择 `.SchDoc`
- 保存为 `.kicad_sch`

### 6.3 与 PCB 区分

PCB 与原理图在 GUI 上应拆成两个独立状态机：

- `PcbGuiImportFlow`
- `SchematicGuiImportFlow`

避免一个流程中混杂多类窗口与分支判断。

## 7. 云端无人值守与 API 设计

### 7.1 运行模型

远程运行环境定义为：

- 云端 Windows 虚拟机
- 持续在线桌面会话
- 已安装 KiCad
- 常驻桌面代理进程

该模型不是纯 headless，而是“无人值守但有桌面代理”。

### 7.2 服务分层

建议分为：

1. `API Service`
2. `GUI Job Worker`
3. `Windows Desktop Agent`

其中：

- API Service 只负责接收任务与查询状态
- Worker 串行消费 GUI 导入任务
- Desktop Agent 实际执行桌面自动化

### 7.3 Job 工作目录

每个任务使用独立目录：

- `jobs/<job_id>/input`
- `jobs/<job_id>/run`
- `jobs/<job_id>/output`
- `jobs/<job_id>/artifacts`

这样可保证：

- 输入隔离
- 输出隔离
- 日志隔离
- 支持失败回放

### 7.4 API 最小集合

建议预留如下接口：

- `POST /jobs/gui-import`
- `GET /jobs/{job_id}`
- `GET /jobs/{job_id}/artifacts`

`POST /jobs/gui-import` 的输入建议包括：

- 文件上传或服务端文件路径
- 文档类型：`pcb` / `schematic`
- 源格式：`altium`
- 目标策略
- 输出目录或目标存储位置

### 7.5 返回信息要求

API 返回必须可供 AI 智能体理解，而不只是“成功/失败”：

- `job_id`
- `status`
- `phase`
- `started_at`
- `finished_at`
- `error_message`
- `last_window_title`
- `artifacts`

失败时还应附带：

- 截图路径
- 弹窗文本
- 自动化日志路径

## 8. 失败处理与回退策略

### 8.1 失败证据

每次失败都必须保留：

- 当前阶段名
- 最后一步动作
- 当前窗口标题
- 当前顶层窗口列表
- 错误弹窗文本
- 截图
- automation log

### 8.2 三层恢复策略

1. 同阶段重试
   - 控件未就绪
   - 菜单未展开
   - 文件对话框未稳定

2. 阶段回退
   - 退回上一窗口，重新打开导入入口

3. 任务级重启
   - KiCad 卡死
   - 窗口树异常
   - 导入后无响应

### 8.3 与其他策略的回退关系

后续可支持以下编排模式：

- 用户显式选 GUI：只执行 GUI
- 用户选官方自动：CLI 失败后切 GUI
- 用户选综合策略：CLI -> GUI -> third-party

但第一阶段不一次性实现完整编排，仅先提供单独 GUI 策略。

## 9. 输入限制与前置判断

### 9.1 PCB 输入格式判断

对 PCB 输入需要显式区分：

- 二进制 `.PcbDoc`
- ASCII `.PcbDoc`

当前已知情况：

- 第三方 PCB 链路不支持 ASCII `.PcbDoc`
- GUI 链路是否支持 ASCII `.PcbDoc` 需要单独验证

因此第一阶段建议：

- 先以二进制 `.PcbDoc` 为主支持对象
- 对 ASCII `.PcbDoc` 做单独前置判断与单独报错或分流

### 9.2 桌面环境前置条件

执行 GUI 自动化前需检查：

- 是否为 Windows
- 是否安装 KiCad
- 是否存在可交互桌面会话
- 是否可创建工作目录
- 是否可写输出目录

不满足时直接快速失败，不进入 GUI 阶段。

## 10. 验证策略

### 10.1 成功验证

至少分三层验证：

1. 任务成功验证
   - 输出文件存在
   - 文件大小大于 0

2. 文件有效性验证
   - `.kicad_pcb` 能被基础文本解析
   - 非空壳模板

3. 业务级基础验证
   - 存在板框或边界对象
   - 存在网络、焊盘、走线等关键对象之一

### 10.2 不在 GUI 自动化内直接完成的验证

以下内容不放在第一阶段 GUI 自动化主流程内：

- 走线悬空检查
- 层间对应一致性检查
- 大规模 DRC 结果优化
- 规则自动修正

这些应作为导入成功后的后处理或质量验证阶段。

## 11. 测试设计

### 11.1 单元测试

对以下内容做单测：

- 阶段状态推进
- 超时与重试策略
- 产物路径收集
- 错误报告格式
- 输入格式前置判断

### 11.2 伪造驱动测试

用 mock driver 模拟：

- 窗口出现
- 菜单点击成功
- 弹窗出现
- 保存完成
- 超时失败

用于验证状态机逻辑，而不依赖真实 KiCad。

### 11.3 集成测试

在真实 Windows + KiCad 环境下跑：

- 真实 `.PcbDoc` 导入
- 导入失败截图保留
- 输出文件有效性检查

第一阶段集成测试应重点覆盖 PCB。

## 12. 实施顺序

建议按以下顺序实施：

1. 新增 GUI 自动化运行时与任务目录管理
2. 新增 `kicad-gui-official` 策略骨架
3. 实现 PCB GUI 导入状态机
4. 实现截图、日志、窗口证据收集
5. 接入 `ConversionService / CLI / Web`
6. 补充 PCB GUI 集成测试
7. 再扩展原理图 GUI 导入
8. 最后预留 API / 云端代理接入点

## 13. 风险与应对

### 13.1 KiCad 版本差异

风险：

- 菜单位置、窗口标题、弹窗文案可能在版本间不同

应对：

- 驱动层集中管理版本差异
- 优先使用控件树与角色识别，不依赖固定文字

### 13.2 GUI 会话不稳定

风险：

- 远程桌面断开、桌面锁屏、焦点被抢占

应对：

- 会话预检查
- 单任务串行
- 失败现场完整保留

### 13.3 图像兜底不稳定

风险：

- 分辨率、缩放、主题变化导致误识别

应对：

- 图像识别只做最后兜底
- 主流程优先 UIA 控件驱动

### 13.4 ASCII/二进制输入差异

风险：

- 不同 `.PcbDoc` 变体支持度不同

应对：

- 前置判断
- 显式分流
- 报错中明确输入类型

## 14. 结论

本方案将 KiCad GUI 自动化定义为：

- 统一接入现有转换服务
- 先实现 PCB，再扩展原理图
- 运行在 Windows 桌面代理之上
- 支持后续 API 化给其他 AI 智能体调用

第一阶段最重要的成功标准不是“支持所有文件”，而是：

- PCB GUI 自动化流程可以稳定跑通
- 失败时可定位、可截图、可回放
- 可与现有策略体系共存

在此基础上，再逐步扩展原理图 GUI 自动化与自动回退编排。
