# Altium ASCII -> KiCad 原理图转换器设计文档

日期：2026-05-09

## 1. 背景与目标

本项目的长期目标是将主流 EDA 工具的设计数据转换为 KiCad 格式，最终支持真实工程使用，并可继续扩展为本地 Web 应用和面向其他 AI 智能体的云端接口。

当前设计文档只覆盖第一个子项目：

`子项目 A：Altium ASCII -> KiCad 原理图转换器`

根据 `Altium_to_KiCad_需求核对表Ver1.0.xlsx` 的当前结论，该子项目的目标不是直接产出可生产的完整 KiCad 工程，而是产出：

- 可在 KiCad 中正常打开并继续编辑的原理图工程
- 可作为后续 PCB 转换的可靠上游输入
- 保留关键工程字段和连接语义
- 对降级处理和不支持对象提供明确报告

## 2. 范围与约束

### 2.1 已确认范围

- 第一阶段输入来源：`Altium Designer`
- 第一阶段输入方式：`Altium ASCII 原理图`
- 当前只覆盖：`原理图`
- 团队确认可以稳定导出并提供 ASCII 文件
- 第一阶段接受“先做原理图，作为后续 PCB 转换上游”
- 第一阶段不要求直接交付“完整可生产 KiCad 工程”
- 第一阶段需要交付 `CLI + 本地 Web`
- 本地 Web 应用应是同一核心引擎的封装
- 实现语言：`Python`

### 2.2 原理图能力边界

- 是否必须支持 `多页原理图`：`待确认`
- 第一阶段不纳入 `复杂层级原理图`
- `sheet symbol`、`bus / bus entry`、`harness`、`重复通道 / channel`、`跨页 port / off-sheet connector`：`待确认`
- 连接语义必须正确，同时图形外观应尽量接近原图

### 2.3 电气语义约束

- `net label` 是明确要求必须正确处理的对象
- `错连 / 断连 / 引脚号错 / 网络名错` 绝对不能接受
- 哪些显示对象允许降级：`待确认`
- 是否存在高优先级关键页面或关键网络：`待确认`

### 2.4 器件与字段策略

- 优先使用 `本地库 / 公司自定义符号库`
- 已确认存在 `公司自定义符号库`
- 是否存在必须映射到指定 KiCad 符号的器件清单：`待确认`
- 仍保留生成 `项目私有符号` 的能力，作为未命中本地库时的回退机制

必须保留的字段：

- `Designator`
- `Value / Comment`
- `Footprint`
- `Part Number`
- `Manufacturer`
- `Supplier`
- 自定义参数

待确认字段约束：

- 是否存在公司内部专用字段必须保留
- 字段命名是否有统一规范

### 2.5 工程可用性要求

- 转换结果必须支持继续修改参数
- 转换结果必须支持继续修改网络
- 转换结果必须支持替换符号
- 转换结果必须支持重新指定封装
- 是否要求从转换后的原理图直接继续推进到 PCB：当前 `不需要`

### 2.6 验收与样例约束

- 需要准备 `3-5` 个真实样例工程作为测试基线
- 是否能提供简单项目、中等复杂项目、含特殊对象项目：`待确认`
- 样例是否需要脱敏：`待确认`
- 验收最看重的指标组合：`待确认`

### 2.7 部署与合规待确认项

- Altium 版本范围
- 云端接口是否只供内部 AI 智能体调用
- 是否要求离线运行
- Python 版本或部署限制
- 数据保密或合规要求

## 3. 项目拆分

为了最终达到“可直接生产的完整 KiCad 工程”，整体项目拆分为以下三个子项目：

### 子项目 A

`Altium ASCII -> KiCad 原理图转换器`

### 子项目 B

`Altium ASCII -> KiCad PCB 转换器`

### 子项目 C

`工程一致性与生产校验`

包括但不限于：

- 原理图/PCB 网络一致性
- 封装关联完整性
- ERC/DRC 前置检查
- 生产输出验证

本设计文档仅定义 `子项目 A`。

## 4. 设计原则

- 使用 `中间模型（IR）`，不采用输入到输出的直通式拼接
- 转换核心与调用方式解耦
- 优先保证工程语义正确，同时在可控范围内尽量保持与原图接近的视觉结果
- 所有降级处理必须显式记录
- 为后续扩展更多 EDA 输入做准备
- 不把待确认事项提前写死到实现假设中
- 自研实现、`KiCad` 官方能力链路、第三方库链路都视为可行实现路径，先实现并对比结果，再决定长期优先级

## 4.1 实现路径策略

第一阶段不预先锁定唯一实现路径，而是并行保留以下三种方案：

### 路径 A：自研转换链路

即当前设计文档的主结构：

- `Altium ASCII` 解析
- 内部 `IR` 归一化
- 本地符号库解析
- KiCad 原理图写出
- 校验与报告

价值：

- 控制力最强
- 最适合承接自定义字段、公司本地库和后续 Web / AI 接口

### 路径 B：KiCad 官方能力辅助链路

这里的含义不是让 KiCad 直接替代全部转换逻辑，而是把 KiCad 官方能力作为一个可执行的候选实现或后处理链路，例如：

- 使用 KiCad 官方文件结构和能力做输出兼容性增强
- 使用 KiCad 自身可用的导入、检查或工程读取能力做辅助转换或验证

价值：

- 更贴近 KiCad 官方生态
- 有助于降低输出格式偏差

### 路径 C：第三方库链路

例如 `Altium2kicad` 或其他可用开源项目。

使用方式包括：

- 作为完整备选转换链路
- 作为部分阶段的能力来源
- 作为对照输出生成器

价值：

- 可快速获得一个可运行基线
- 有助于识别自研链路遗漏的对象和规则

### 路径选择原则

- 当前阶段 `不设定固定优先级`
- 三种方法都应具备接入同一评测框架的能力
- 在完成可运行版本后，对三种方法进行统一质量对比
- 质量对比结果将作为后续长期主路径选择依据

### 统一对比维度

后续至少按以下维度比较三种方法：

- 工程可打开性
- 可编辑性
- `net label` 和网络语义正确性
- 字段保留完整度
- 本地符号库适配能力
- 图形还原质量
- 对异常输入的稳定性
- 二次扩展成本

## 5. 总体架构

建议采用 `Core Engine + Adapters` 结构。

### 5.1 Parser

职责：

- 读取 `Altium ASCII` 原理图文件
- 解析为 Altium 侧原始对象集合
- 保留足够的原始结构用于调试和回溯

约束：

- 这一层不做 KiCad 相关决策
- 这一层只负责“忠实读取”

### 5.2 Normalizer / IR Builder

职责：

- 将 Altium 原始对象转换为内部统一的 `IR`
- 归一化坐标、旋转、字段和连接关系

价值：

- 隔离输入格式差异
- 为未来接入 `Eagle`、`EasyEDA`、`OrCAD` 等工具保留统一语义层

### 5.3 Symbol Resolver

职责：

- 优先将器件映射到 `本地库 / 公司自定义符号库`
- 本地库未命中时，再按规则决定是否映射到公共库或生成项目私有符号
- 为后续补充“指定器件必须映射到指定符号”的规则预留接口

输出：

- 最终符号决策结果
- 符号映射报告
- 未命中本地库的回退清单

### 5.4 KiCad Writer

职责：

- 将 IR 和符号决策结果写出为 KiCad 工程文件

第一阶段至少输出：

- `.kicad_pro`
- `.kicad_sch`
- 多页相关原理图文件
- 项目私有 `.kicad_sym`
- 转换报告

### 5.5 Application Layer

职责：

- 暴露调用入口

第一阶段：

- `CLI`
- `本地 Web/API`

后续：

- 云端 `HTTP API`

### 5.6 Shared Services

职责：

- 日志
- 配置
- 错误与告警收集
- 路径与输出管理
- 转换报告写出

### 5.7 Conversion Strategy Layer

职责：

- 抽象三类转换路径的统一接口
- 屏蔽具体实现来自自研、KiCad 官方能力还是第三方库
- 为统一测试、统一报告和统一质量对比提供稳定边界

建议接口：

- `convert(input) -> conversion artifacts`
- `validate(artifacts) -> conversion report`
- `describe() -> strategy metadata`

## 6. 内部 IR 设计

IR 的目标不是复制 Altium 文件结构，而是表达跨 EDA 成立的原理图语义。

### 6.1 Project

表示一次转换后的原理图工程。

包含：

- 工程 ID / 名称
- 输入文件列表
- 页集合
- 全局元数据
- 转换告警集合

### 6.2 Sheet

表示单张原理图页。

包含：

- 页 ID
- 页名
- 页序号
- 页面尺寸 / 原点 / 坐标系
- 页内对象集合
- 页级属性

说明：

- 是否需要完整支持多页原理图仍待确认，因此 `Sheet` 模型必须从一开始支持单页和多页两种模式
- 复杂层级关系当前不进入第一阶段 IR 范围

### 6.3 ComponentInstance

表示器件实例，而不是符号模板。

包含：

- 实例 ID
- `Designator`
- `Value/Comment`
- 符号引用键
- 放置位置
- 旋转 / 镜像
- 字段集合
- 引脚实例列表
- 目标封装名

### 6.4 SymbolDefinition

表示抽象后的符号定义。

包含：

- 定义 ID
- 显示名
- 图形元素
- 引脚定义
- 默认字段模板

来源：

- 映射到本地库或公司自定义库
- 从 Altium 元件生成项目私有符号

### 6.5 Pin

表示引脚语义。

包含：

- 引脚号
- 引脚名
- 电气类型
- 图形方向 / 位置

### 6.6 Net

表示逻辑网络。

包含：

- 网络 ID
- 网络名
- 连接点集合
- 参与对象引用

说明：

- 不能只依赖“画出来的线段”
- 必须保留逻辑连接语义

### 6.7 Wire / Junction / Label / Port

这些对象应独立于 `Net` 保留：

- `WireSegment`
- `Junction`
- `NetLabel`
- `Port`

原因：

- KiCad 原理图既要求逻辑网络，也要求可编辑图形对象
- 当前已明确 `NetLabel` 必须正确处理
- `Port` 的第一阶段支持范围仍待样例工程确认

### 6.8 GraphicPrimitive

表示非电气图形对象。

包括：

- 线
- 矩形
- 圆
- 多边形
- 文本

### 6.9 Field

统一表示器件属性。

内部应同时保留：

- 标准字段名
- 原始字段名

第一阶段至少标准化：

- `Designator`
- `Value/Comment`
- `Footprint`
- `Part Number`
- `Manufacturer`
- `Supplier`
- 自定义参数

## 7. 转换流程

### 7.0 多策略执行模型

完整系统应支持以下执行模式：

1. 单策略执行
   - 指定使用 `自研`、`KiCad 官方能力辅助链路` 或 `第三方库链路`

2. 多策略对比执行
   - 对同一输入运行多个策略
   - 输出多份结果和统一比较报告

第一阶段至少要保证架构允许这两种模式并存，即使最早只完整落地其中一条链路。

### 7.1 Input Loader

职责：

- 读取输入文件集合
- 建立页顺序和工程上下文
- 执行基础完整性检查

检查项：

- 文件是否齐全
- 页是否可识别
- 编码是否可解析
- 是否存在明显缺页或引用断裂

### 7.2 Altium Parser

职责：

- 将输入文件解析为 Altium 原始对象树

### 7.3 Normalizer -> IR

职责：

- 坐标和旋转归一化
- 实例与定义拆分
- 连接关系归并
- 字段标准化
- 单页或多页工程组织

### 7.4 Symbol Resolution

流程：

1. 尝试命中 `本地库 / 公司自定义符号库`
2. 命中失败则按规则映射公共库或生成项目私有符号

同时输出：

- 命中的规则和原因
- 未命中的列表
- 回退私有符号的列表
- 不确定字段的告警

### 7.5 KiCad Project Writer

职责：

- 写出 KiCad 工程文件
- 组织单页或多页原理图文件
- 写出项目私有符号库
- 写出转换报告

### 7.6 Validation Pass

写出完成后立即执行程序内校验。

至少检查：

- 页数是否一致
- 器件数是否一致
- 网络数是否基本一致
- `net label` 解析与落点是否一致
- 未解析对象是否记录
- 未命中本地库的对象是否记录
- 必需字段是否丢失

## 8. 子项目 A 的责任边界

### 8.1 明确负责

- 原理图页结构
- 器件与符号
- 位号与常见字段
- 网络与连线语义
- `net label` 正确转换
- `Footprint` 等关键字段保留
- 工程级转换报告生成

### 8.2 明确不负责

- `.PcbDoc -> .kicad_pcb` 转换
- 板框、走线、过孔、铺铜、设计规则转换
- Gerber、钻孔、装配文件等生产输出
- 最终制造可行性判定
- 复杂层级原理图支持

## 9. 与后续子项目的接口

### 9.1 面向 PCB 转换

需要稳定输出：

- 工程元数据
- 器件标识
- `Footprint` 字段
- 网络命名约定

### 9.2 面向生产校验

需要稳定输出：

- 转换报告
- 丢失对象清单
- 符号映射结果
- 一致性检查结果

### 9.3 面向多策略质量评估

需要稳定输出：

- 生成产物路径
- 关键对象统计
- 校验结果
- 质量评分输入项

## 10. 错误处理策略

### 10.1 Fatal

无法继续生成可用 KiCad 工程，必须终止。

例如：

- 输入损坏或不是预期的 Altium ASCII
- 页结构无法解析
- 核心连接语义丢失到无法恢复
- 输出文件结构不合法
- 出现 `错连 / 断连 / 引脚号错 / 网络名错`

### 10.2 Recoverable With Warning

可以继续生成工程，但存在降级处理。

例如：

- 某些图形对象无法完全等价表达
- 某些符号无法命中本地库，只能回退为项目私有符号
- 某些非关键字段保留不完整
- 某些装饰性文本或样式降级

要求：

- 必须记录到转换报告
- 不允许静默吞掉

### 10.3 Informational

记录对排查和审计有价值的信息。

例如：

- 本地库映射成功
- 字段被标准化重命名
- 坐标被归一化调整

## 11. 兼容策略

第一阶段明确支持：

- `Altium ASCII` 原理图
- `单页原理图`
- 常见基础对象：
  - 器件
  - 引脚
  - 连线
  - 网络标签
  - 文本
  - 常见图形
- 常见工程字段：
  - `Designator`
  - `Comment/Value`
  - `Footprint`
  - `Part Number`
  - `Manufacturer`
  - `Supplier`
  - 自定义参数
- 本地库优先映射，失败回退私有符号

第一阶段待样例确认后再决定是否纳入：

- `多页原理图`
- `端口`
- `bus / bus entry`
- `harness`
- `重复通道 / channel`
- `跨页 port / off-sheet connector`

第一阶段不承诺：

- 所有 Altium 特有对象 100% 等价
- 复杂层级原理图支持
- 所有厂商自定义字段语义都能自动理解
- 与后续 PCB 100% 自动零修正衔接

## 12. 测试策略

### 12.1 Parser Tests

目标：

- 验证 Altium ASCII 片段能被正确解析
- 防止语法解析回归

### 12.2 Normalizer / IR Tests

目标：

- 验证器件、引脚、网络归一化正确
- 验证单页与多页组织逻辑正确
- 验证字段标准化正确

### 12.3 Writer Tests

目标：

- 验证固定 IR 能稳定写出 KiCad 文件
- 验证文件结构和字段完整性

### 12.4 Golden Project Tests

准备真实 Altium 示例工程，执行端到端转换并核对关键指标：

- 页数
- 器件数
- 网络数
- `net label` 一致性
- 未映射符号数
- 告警数
- 关键字段保留情况

建议样例规模：

- `3-5` 个真实项目
- 至少覆盖一个简单项目
- 复杂度更高和特殊对象样例待确认后补齐

### 12.5 多策略对比测试

对同一 Golden 输入，分别运行：

- 自研转换链路
- `KiCad` 官方能力辅助链路
- 第三方库链路

至少比较：

- 生成是否成功
- KiCad 是否可打开
- `net label` 与网络语义是否正确
- 必保字段是否保留
- 本地符号库映射表现
- 图形与布局保真度
- 告警与错误分布

### 12.6 后续可选集成测试

如果环境允许，应增加：

- KiCad 打开验证
- 非交互式工程校验

## 13. 验收标准

`子项目 A` 完成时，至少满足以下条件：

- 对约定范围内的 Altium ASCII 原理图可稳定生成 KiCad 工程文件
- 输出工程可在 KiCad 中打开并继续编辑
- 已确认纳入范围的页结构被正确保留
- 器件位号、数值、常见字段正确保留
- `Designator`、`Value`、`Footprint`、`Part Number`、`Manufacturer`、`Supplier`、自定义参数被按规则保留
- 网络连接语义基本正确，无系统性错连或断连
- `net label` 转换正确
- 本地库映射与私有符号回退机制可用
- 每次转换都生成机器可读报告
- 遇到不支持对象时会显式告警
- 具备 `3-5` 个真实样例工程作为回归基线
- 三种实现路径可以在统一样例集上产出可比较结果

## 14. 成功定义

`子项目 A` 的成功标准不是“把文件转出来”，而是：

- 生成的 KiCad 原理图工程可继续用于真实工程工作
- 转换系统清楚记录哪些内容被正确转换
- 对降级、不支持和不确定项给出明确结果
- 转换结果允许继续修改参数、网络、符号和封装
- 为后续 PCB 转换和生产校验提供可靠上游输入
- 对自研、KiCad 官方能力链路、第三方库链路有统一质量对比依据
