# KiCad GUI 自动化实现计划

> 基于已确认的《KiCad GUI 自动化设计文档》执行。  
> 实施顺序遵循：**先 PCB GUI 导入跑通，再扩展原理图 GUI 导入，同时从一开始预留云端 Windows 桌面代理与 API 调用面。**

---

## 1. 实施目标

本轮实现的目标不是“一次性做完整个平台”，而是先把最关键的主链路做稳定：

1. 新增 `kicad-gui-official` 策略
2. 实现 `Altium .PcbDoc -> KiCad GUI 导入 -> .kicad_pcb 保存`
3. 接入现有 `ConversionService / CLI / Web`
4. 保留 GUI 自动化日志、截图和失败现场
5. 为后续 `.SchDoc` GUI 导入与 API 化预留稳定边界

### 本轮验收标准

- 能通过统一调用面选择 GUI 自动化策略
- 对受支持的 `.PcbDoc` 输入，能自动生成 `.kicad_pcb`
- 失败时能返回明确阶段、错误信息、日志和截图路径
- 自动化逻辑有可测试的状态机与 mock 测试
- 不破坏现有 `custom / kicad-official / third-party` 行为

### 本轮不强求完成

- 原理图 GUI 导入全量落地
- 云端 API 服务正式部署
- 多任务并发调度
- 复杂图像识别自动恢复
- GUI 导入后的高级 DRC 与质量修复

## 4. 阶段 A：GUI 自动化基础设施

### A1. 依赖与运行时约定

需要在 `pyproject.toml` 中增加 GUI 自动化依赖：

- `pywinauto`
- `psutil`
- 可选兜底：
  - `Pillow`
  - `pyautogui` 或替代截图/点击工具

说明：

- 主驱动优先使用 `pywinauto/UIA`
- 图像识别或坐标点击只作为局部兜底能力，不作为主方案

### A2. 桌面会话检查

在 `session.py` 中实现：

- 是否为 Windows
- 是否存在可交互桌面会话
- 是否存在 KiCad 可执行文件
- 是否允许创建任务目录

建议暴露：

- `assert_gui_environment_ready()`
- `create_job_workspace()`
- `acquire_gui_job_lock()`

### A3. 任务目录结构

每次 GUI 导入任务目录统一为：

- `outputs/gui-jobs/<job_id>/input`
- `outputs/gui-jobs/<job_id>/run`
- `outputs/gui-jobs/<job_id>/output`
- `outputs/gui-jobs/<job_id>/artifacts`

这样可以保证：

- 输入输出隔离
- 故障回放容易
- 日志与截图归档清晰

### A4. 运行时上下文

在 `runtime.py` 中建立：

- 当前阶段
- 当前动作
- 日志记录器
- 截图目录
- 窗口快照采集能力

建议形成一个运行时对象，例如：

- `GuiAutomationRuntime`

核心能力包括：

- `set_phase(...)`
- `log_step(...)`
- `capture_screenshot(...)`
- `dump_windows(...)`
- `record_failure(...)`

### A5. 本阶段测试

新增单元测试：

- `tests/gui/test_session.py`
- `tests/gui/test_runtime.py`

验证内容：

- 任务目录创建
- 阶段记录
- 日志文件生成
- 截图/窗口 dump 路径生成逻辑
- 环境检查异常抛出逻辑

## 5. 阶段 B：KiCad GUI 通用驱动

### B1. 启动与连接 KiCad

在 `driver.py` 中实现：

- 启动 KiCad 主程序
- 等待主窗口出现
- 连接主窗口句柄
- 检测卡死/崩溃恢复弹窗

建议提供：

- `launch_kicad()`
- `wait_main_window()`
- `close_kicad()`

### B2. 通用窗口与菜单操作

能力范围：

- 根据标题/角色查找窗口
- 点击菜单项
- 填写文件路径
- 点击确认/保存/关闭弹窗

注意事项：

- 菜单文案不要在高层写死
- 版本差异尽量收敛在驱动层

### B3. 图像兜底点

只有以下场景才允许图像兜底：

- UIA 无法识别导入按钮
- 某些文件对话框控件树异常
- 保存确认按钮找不到

第一版不做 OCR 复杂识别，只做：

- 模板图点击
- 关键区域截图留证

### B4. 本阶段测试

先以 mock 测试为主，不强依赖真实 KiCad：

- 驱动方法调用顺序
- 等待超时逻辑
- 菜单失败重试逻辑

## 6. 阶段 C：PCB GUI 导入状态机

### C1. 状态机定义

在 `pcb_import.py` 中实现：

- `launch_kicad`
- `wait_main_window`
- `open_import_menu`
- `select_input_file`
- `confirm_import`
- `wait_import_complete`
- `save_output`
- `collect_artifacts`

### C2. 输入前置判断

执行 GUI PCB 导入前增加：

- 输入文件存在检查
- 文件后缀检查
- ASCII / 二进制 `.PcbDoc` 区分

第一阶段建议策略：

- 以二进制 `.PcbDoc` 为主支持对象
- 对 ASCII `.PcbDoc` 显式报“不支持 GUI 官方导入”或“暂未验证”

### C3. 导入完成判定

不要靠固定时间等待，应组合判断：

- 编辑器窗口已加载目标工程
- 窗口标题变化
- 导入弹窗已关闭
- 当前保存操作可执行

### C4. 保存输出

统一将结果保存到任务工作目录下，再回写到调用方输出目录。

标准输出：

- `<project>.kicad_pcb`
- 可选 `<project>.kicad_pro`

若 KiCad GUI 未生成 `.kicad_pro`，则沿用现有最小工程文件补齐逻辑。

### C5. 本阶段测试

新增：

- `tests/gui/test_pcb_import.py`

测试方式：

- mock driver 测状态机
- mock artifacts 测输出收集
- mock failure 测错误报告

## 7. 阶段 D：策略接入与统一调用面

### D1. 新增 GUI 策略

新增：

- `src/eda2kicad/strategies/kicad_gui_official.py`

建议策略 ID：

- `kicad-gui-official`

策略职责：

- 接收输入路径
- 根据后缀决定走 PCB GUI 或原理图 GUI
- 组织运行时
- 读取输出文件
- 回填 `ConversionPayload`

### D2. 与现有 `ConversionPayload` 对齐

成功时：

- `board_text`
- `board_extension`
- `auxiliary_text_artifacts`
- `report`

失败时：

- 抛出包含阶段和原因的 `ValueError`
- 附带日志/截图路径写入报告

### D3. 接入 `registry.py`

将 GUI 策略注册进现有策略表，保证：

- CLI 下可选
- Web 下可选
- Service 下可选

### D4. 接入 CLI

CLI 保持现有风格，只扩展：

- `--strategy kicad-gui-official`

如需后续更清晰，可再补：

- `--gui-timeout`
- `--gui-artifacts-dir`

但第一版不强求暴露过多参数。

### D5. 接入 Web

Web 只负责：

- 让用户可选 GUI 策略
- 展示成功/失败结果
- 展示生成文件路径

不在第一版里直接展示截图缩略图或复杂任务状态页。

### D6. 本阶段测试

补充：

- `tests/test_service.py`
- `tests/test_cli.py`
- `tests/web/test_app.py`

验证内容：

- 可选择 GUI 策略
- 结果文件被正确写出
- 失败时错误信息被返回

## 8. 阶段 E：失败证据与质量验证

### E1. 失败现场最小集合

每次失败必须落地：

- `automation.log`
- `failure-screenshot.png`
- `window-dump.json`
- `phase.txt` 或 report 内阶段字段

### E2. 报告结构补齐

建议在 `report.json` 中新增：

- `automation.phase`
- `automation.last_action`
- `automation.screenshot`
- `automation.window_dump`

### E3. 成功后的最小有效性检查

对 `.kicad_pcb` 做最小有效性检查：

- 文件存在
- 大小大于 0
- 文本包含 `(kicad_pcb`

有条件时再补：

- 是否存在板框对象
- 是否存在焊盘/走线/网络

### E4. 本阶段测试

mock 测试为主，真实 GUI smoke test 为辅。

## 10. API 与云端代理预留

本轮不直接交付 API 服务，但实现中必须预留这些边界：

- 任务目录独立
- 运行时信息结构化
- 失败证据可文件化
- 策略调用不依赖 Web 或 CLI 环境

后续 API 只需在上层包装：

- `POST /jobs/gui-import`
- `GET /jobs/{job_id}`
- `GET /jobs/{job_id}/artifacts`

因此当前实现中要避免：

- 在 GUI 驱动里写死 Web/CLI 逻辑
- 在驱动层直接依赖终端输出格式

## 12. 主要风险与应对

### 风险 1：KiCad 版本差异导致控件路径变化

应对：

- 将控件查找逻辑集中到驱动层
- 允许多套 locator 回退

### 风险 2：远程桌面会话不稳定

应对：

- 任务前环境检查
- 失败时立即记录窗口与截图

### 风险 3：ASCII `.PcbDoc` 支持不确定

应对：

- 前置判断
- 明确报错
- 不把 GUI 自动化失败和文件格式失败混在一起

### 风险 4：图像兜底引入脆弱性

应对：

- 只在 UIA 无法工作的局部点使用
- 主流程坚持 UIA 优先

## 13. 完成判定

当以下条件满足时，可认为第一阶段完成：

- `kicad-gui-official` 已接入统一调用面
- PCB GUI 导入可在本机 Windows + KiCad 环境下跑通
- 失败有日志、截图、阶段信息
- 自动化逻辑具备 mock 测试覆盖
- 原有 3 种策略功能不回退

## 14. 建议的下一个执行会话

实际编码时建议按以下顺序推进：

1. 先建 `gui/` 基础设施和测试骨架
2. 只打通 PCB GUI import 的 happy path
3. 再补失败证据与回退
4. 最后再扩原理图 GUI 导入

如果使用子智能体并行执行，建议拆分为：

- Worker A：GUI 基础设施与 runtime
- Worker B：PCB GUI 状态机与 driver
- Worker C：策略接入与 service/cli/web 集成
