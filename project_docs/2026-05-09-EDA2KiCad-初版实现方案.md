# EDA 转 KiCad 实现方案

## 1. 项目目标

本项目目标是实现一个将主流 EDA 设计数据转换为 KiCad 工程的转换框架，分阶段落地：

- 第一阶段：`Altium ASCII 原理图 -> KiCad 原理图`
- 第二阶段：`Altium PCB 数据 -> KiCad PCB`
- 第三阶段：统一质量校验、生产前检查和多工具扩展

当前已确认的首个落地方向是：

- 输入来源：`Altium Designer`
- 输入方式：`Altium ASCII`
- 当前重点：`原理图转换`
- 交付形态：`CLI + 本地 Web`

## 2. 总体实现思路

整体采用“`统一转换核心 + 多策略实现 + 统一评测`”的架构。

核心思想是：

1. 先把 EDA 输入解析为统一内部模型 `IR`
2. 再从 `IR` 输出为 KiCad 格式
3. 同时保留三种实现路径并行验证：
   - `自研转换链路`
   - `KiCad 官方能力辅助链路`
   - `第三方库链路（如 Altium2kicad）`
4. 三种方法都跑同一批样例工程，对比质量后再决定长期主路径

## 3. EDA 转 KiCad 的核心流程

### 3.1 原理图转换流程

`输入文件 -> 文件加载 -> Altium ASCII 解析 -> 内部 IR 归一化 -> 符号解析/字段映射 -> KiCad 原理图写出 -> 校验 -> 转换报告输出`

关键步骤包括：

- 读取原理图对象：器件、引脚、线段、网络标签、字段、图形对象
- 归一化为统一模型：器件实例、符号定义、网络、连线、字段
- 将 Altium 器件优先映射到本地/公司符号库
- 未命中时回退生成私有符号
- 写出 `.kicad_sch`、必要符号库和 `report.json`
- 检查关键语义是否正确，例如 `net label`、引脚号、网络名

### 3.2 后续 PCB 转换流程

`PCB 输入 -> 板层/器件/网络/走线/过孔/规则解析 -> 内部 PCB IR -> KiCad PCB 写出 -> 设计规则覆写/校验 -> 对比报告输出`

关键步骤包括：

- 解析板框、层栈、器件封装、走线、过孔、铺铜、网络
- 恢复层间关系和坐标系统
- 保留封装绑定和网络一致性
- 尽量读取并转译原始设计规则
- 输出 `.kicad_pcb` 及相应校验结果

## 4. 输入输出标准

### 4.1 当前阶段输入标准

- 输入工具：`Altium Designer`
- 输入格式：`Altium ASCII 原理图`
- 输入内容至少应包含：
  - 器件定义
  - 引脚信息
  - 线段与连接关系
  - `net label`
  - 常用字段：`Designator`、`Value`、`Footprint`、`Part Number`、`Manufacturer`、`Supplier`

### 4.2 当前阶段输出标准

输出结果应至少包括：

- `KiCad 原理图文件`
- 必要时的项目私有符号库
- 机器可读转换报告 `report.json`

输出质量标准：

- 能在 KiCad 中正常打开
- 能继续编辑参数、网络、符号、封装
- `net label` 正确
- 不允许出现系统性错连、断连、引脚号错误、网络名错误

### 4.3 后续 PCB 阶段输入标准

后续 PCB 转换阶段，输入应尽可能包含：

- 板层定义
- 器件与封装位置
- 网络关系
- 走线、过孔、铺铜
- 设计规则
- 板配置参数

### 4.4 后续 PCB 阶段输出标准

输出结果至少应满足：

- `.kicad_pcb` 可被 KiCad 正常打开
- 层间对应关系正确
- 网络与封装关系正确
- 走线、过孔、铺铜结构正确
- 可进入后续 DRC/生产准备流程

## 5. 当前已知重点问题

### 5.1 原理图阶段问题

1. `Altium ASCII` 结构不稳定

 不同版本可能存在字段差异、对象差异、命名差异

2. 符号库映射困难

- 公司本地符号库、自定义命名和 KiCad 目标库未必一一对应

3. 网络语义容易出错

- 错连、断连、引脚号错、网络名错都是不可接受的问题


### 5.2 PCB 阶段问题

1. `PCB 走线不能悬空`

- 任何走线断点、孤立线段、无效连接都会直接影响工程可用性

2. `层间对应不能有误`

- 顶层、底层、内层、过孔和层栈映射错误会导致板级数据失真

3. `设计规则难以还原`

- 如果原始间距、线宽、过孔、禁布区规则丢失，导入 KiCad 后可能出现大量 DRC 问题

4. `电路板配置问题`

- 板厚、层叠、网络类、约束参数、区域设置等如果丢失，会影响工程继续设计和生产准备

## 6. 暂定解决方案

### 6.1 原理图阶段

1. 采用统一 `IR`

- 不直接做“Altium 输入 -> KiCad 输出”的硬拼接
- 先抽象成统一语义模型，方便后续扩展更多 EDA

2. 建立本地符号库优先机制

- 优先映射公司本地库
- 未命中时回退生成私有符号
- 后续再引入精细映射规则

3. 强化关键语义校验

- 把 `net label`、引脚号、网络名、一致性检查放入强校验
- 对不可接受错误直接判定失败

4. 建立多策略并行对比

- 自研链路、KiCad 官方能力链路、第三方库链路同时进入评测
- 通过样例工程比较输出质量，而不是先凭经验选路线

### 6.2 PCB 阶段

1. 增加 PCB 专用 IR

- 单独抽象板层、走线、过孔、铺铜、规则、板配置

2. 将“悬空走线”列为强约束校验项

- 转换后自动检查是否存在孤立线段、未闭合连接、无效终点

3. 将“层间对应正确”列为强约束校验项

- 转换前后做层映射表
- 对层号、层名、通孔/盲埋孔关系进行检查

4. 优先读取原始设计规则

- 如果源文件能读取到规则，则优先转换并覆写到 KiCad 工程
- 目标是减少导入后大量间距、线宽、过孔相关告警

5. 引入规则映射与覆写层

- 先从 Altium PCB 中提取可明确识别的规则项，例如 `net class`、线宽、间距、过孔尺寸、差分对、禁布区和层约束
- 将源规则转换成一张可追踪的映射表，再写入 KiCad 的 `Board Setup`、`Net Class` 和可用的 DRC 规则配置
- 只对“有明确对应关系”的规则做覆写，无法稳定映射的规则保留默认值并在报告中标注
- 不做全局关闭 DRC，只对已映射且可信的规则项做定向放宽，避免掩盖真实工程问题

6. 保留电路板配置

- 尽量同步板框、层栈、板厚、网络类、区域和约束参数

## 7. 三种实现路径的定位

### 7.1 自研链路

优点：

- 控制力最强
- 最适合承接公司库、自定义字段和后续接口能力

风险：

- 开发周期较长
- 初期覆盖面可能不如成熟项目

### 7.2 KiCad 官方能力辅助链路

优点：

- 更贴近 KiCad 官方格式和生态
- 适合作为输出兼容性增强和验证辅助

风险：

- 官方能力未必覆盖完整转换需求
- 与自定义流程整合成本需要验证

### 7.3 第三方库链路

优点：

- 可快速得到一个可运行基线
- 有利于识别自研链路遗漏的对象和规则

风险：

- 可维护性受外部项目影响
- 与本地库、自定义字段、长期产品化目标可能不完全匹配

## 8. 质量评估标准

三种方法后续统一按以下维度进行对比：

- KiCad 是否可打开
- 原理图/PCB 是否可继续编辑
- `net label` 和网络语义是否正确
- 字段保留是否完整
- 符号/封装映射是否合理
- 图形和布局保真度
- PCB 走线是否悬空
- 层间对应是否正确
- 设计规则是否能正确继承或覆写
- 电路板配置是否保留
- 稳定性和扩展成本

## 9. 当前阶段结论

当前建议按以下策略推进：

- 先完成 `Altium ASCII -> KiCad 原理图` 的最小可运行版本
- 同时预留并接入 `KiCad 官方能力链路` 和 `第三方库链路`
- 用统一样例集做质量对比，不提前锁定唯一主路径
- 在原理图阶段打通后，再进入 PCB 转换和生产级校验

这样做的好处是：

- 先尽快形成可运行结果
- 同时避免过早押注单一路线
- 为后续 PCB 转换和工程化落地保留主动权
